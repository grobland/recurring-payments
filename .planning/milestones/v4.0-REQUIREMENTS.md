# Requirements: Recurring Payment Intelligence

**Defined:** 2026-03-17
**Core Value:** Users upload bank/credit-card statement PDFs and the system automatically detects, tracks, and manages all recurring payments — subscriptions, bills, utilities, loans, and more — with minimal manual effort.

## v4.0 Requirements

### Schema & Domain Model

- [x] **SCHEMA-01**: `recurring_kind` enum with values: subscription, utility, insurance, loan, rent_mortgage, membership, installment, other_recurring
- [x] **SCHEMA-02**: `recurring_status` enum with values: active, paused, cancelled, dormant, needs_review
- [x] **SCHEMA-03**: `amount_type` enum with values: fixed, variable
- [x] **SCHEMA-04**: `merchant_entities` table — canonical merchant identity with name, category, website, known aliases
- [x] **SCHEMA-05**: `merchant_aliases` table — maps statement descriptors to merchant entities, with trigram index for fuzzy matching
- [x] **SCHEMA-06**: `raw_statement_lines` table — immutable raw extracted lines from PDFs preserving original text
- [x] **SCHEMA-07**: Refactor `transactions` table — add normalized_description, source_hash for deduplication, remove subscription-specific columns
- [x] **SCHEMA-08**: `recurring_series` table — algorithmically detected stream of related transactions with cadence, amount stats, confidence
- [x] **SCHEMA-09**: `recurring_series_transactions` junction table — links transactions to series with match_confidence
- [x] **SCHEMA-10**: `recurring_masters` table — user-facing recurring record with all required fields (recurring_kind, status, importance_rating, amount expectations, confidence, billing frequency, etc.)
- [x] **SCHEMA-11**: `recurring_master_series_links` table — links masters to series (one master may have multiple series over time)
- [x] **SCHEMA-12**: `user_transaction_labels` table — user decisions on transactions (recurring, not recurring, ignore) that influence future matching
- [x] **SCHEMA-13**: `review_queue_items` table — items awaiting user review with type, confidence, suggested action
- [x] **SCHEMA-14**: `recurring_events` table — audit trail for recurring master lifecycle events (created, linked, amount changed, paused, etc.)
- [x] **SCHEMA-15**: UUID primary keys on all new tables
- [x] **SCHEMA-16**: Trigram indexes on description columns and alias tables for fuzzy matching
- [x] **SCHEMA-17**: Source hash uniqueness constraint for transaction deduplication

### Ingestion Pipeline

- [x] **INGEST-01**: PDF upload with file type and size validation
- [x] **INGEST-02**: Statement metadata extraction (date range, account info if available)
- [x] **INGEST-03**: Line item extraction preserving raw text in raw_statement_lines
- [x] **INGEST-04**: Normalize raw lines into canonical transactions with normalized_description
- [x] **INGEST-05**: Calculate source_hash for each transaction to prevent duplicates on re-upload
- [x] **INGEST-06**: Parser abstraction interface supporting future provider-specific parsers
- [x] **INGEST-07**: Generic parser implementation using existing OpenAI extraction as base
- [x] **INGEST-08**: Ignore transfers and refunds where detectable
- [x] **INGEST-09**: PDFs stored temporarily, structured data persisted permanently
- [x] **INGEST-10**: Pipeline runs as ordered steps: upload → parse → extract → normalize → deduplicate → resolve merchants → detect recurrence → link masters → generate review items

### Merchant Resolution

- [x] **MERCH-01**: Normalize statement descriptors (lowercase, strip suffixes, remove payment processor prefixes)
- [x] **MERCH-02**: Match transactions to merchant_entities via merchant_aliases
- [x] **MERCH-03**: Fuzzy matching using trigram similarity for unmatched descriptors
- [x] **MERCH-04**: Create new merchant entities for unknown merchants
- [ ] **MERCH-05**: User can create/edit merchant aliases via Settings UI

### Recurrence Detection Engine

- [x] **DETECT-01**: Rule A — known alias hit (exact match to existing recurring master via alias)
- [x] **DETECT-02**: Rule B — fixed monthly recurrence (same amount ±tolerance, similar day-of-month)
- [x] **DETECT-03**: Rule C — variable monthly recurrence (varying amounts within band, similar timing)
- [x] **DETECT-04**: Rule D — annual recurrence (same merchant, ~365 day interval)
- [x] **DETECT-05**: Support weekly, quarterly, and custom cadence detection
- [x] **DETECT-06**: Distinguish fixed vs variable amount_type automatically
- [x] **DETECT-07**: Predict next expected payment date from detected cadence
- [x] **DETECT-08**: Create recurring_series for each detected pattern
- [x] **DETECT-09**: Log detection decisions in recurring_events for audit trail
- [x] **DETECT-10**: Skip clear one-off transactions (no recurring candidates for single occurrences)

### Auto-Linking & Matching

- [x] **LINK-01**: Match order: (1) exact alias-to-master, (2) canonical merchant + compatible cadence/amount, (3) fuzzy descriptor + pattern, (4) create new series candidate, (5) add to review queue if ambiguous
- [x] **LINK-02**: Auto-link when confidence >= 0.85
- [x] **LINK-03**: Add to review queue when confidence 0.60-0.84
- [x] **LINK-04**: Remain unmatched/candidate when confidence < 0.60
- [x] **LINK-05**: User manual decisions override automation and influence future matching
- [x] **LINK-06**: Preserve existing recurring links when re-processing statements
- [x] **LINK-07**: Support descriptor changes (statement text changes but same recurring item)
- [x] **LINK-08**: Support amount changes (price increases/decreases on same recurring item)
- [x] **LINK-09**: Support account/card migration (same recurring item appearing on different account)

### APIs

- [x] **API-01**: Accounts — POST, GET, PATCH /accounts
- [x] **API-02**: Statements — POST, POST /{id}/process, GET, GET /{id}, GET /{id}/transactions
- [x] **API-03**: Transactions — GET, GET /{id}, POST /{id}/label
- [x] **API-04**: Recurring Series — GET, GET /{id}, POST /{id}/confirm, POST /{id}/ignore
- [x] **API-05**: Recurring Masters — GET, GET /{id}, POST, PATCH /{id}, POST /{id}/merge, POST /{id}/status
- [x] **API-06**: Review Queue — GET, POST /{id}/resolve
- [x] **API-07**: Dashboard — GET /dashboard/summary (active counts, totals, upcoming, needs-review)

### UI Screens

- [x] **UI-01**: Dashboard — active recurring/subscription counts, monthly totals, upcoming payments, needs-review alerts, amount change warnings
- [x] **UI-02**: Statement uploads — upload PDF, select account, processing status indicator, statement history
- [x] **UI-03**: Statement detail — extracted rows, inferred merchant, recurring status, linked master, user action buttons
- [x] **UI-04**: Transactions explorer — filters for account, date, amount, merchant, recurring-only, unmatched-only
- [x] **UI-05**: Review queue — suggested recurring items with confidence, recent transactions, confirm/link/ignore/not-recurring actions
- [x] **UI-06**: Recurring master list — tabs/filters for all, subscriptions, bills, needs-review, paused/cancelled
- [x] **UI-07**: Recurring master detail — metadata, linked history, series chain, event log, edit controls
- [x] **UI-08**: Settings/rules — merchant alias management, tolerance settings

### Security & Privacy

- [x] **SEC-01**: Validate upload file type (PDF only) and size limits
- [x] **SEC-02**: Uploaded PDFs stored temporarily, not permanently
- [ ] **SEC-03**: Audit trail for manual user decisions via user_transaction_labels and recurring_events
- [x] **SEC-04**: Secrets and config kept out of source code
