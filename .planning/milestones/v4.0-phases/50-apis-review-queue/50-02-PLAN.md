1. Create `src/app/api/recurring/review-queue/route.ts` — GET:
   - Auth check
   - Query params: itemType (optional filter: "new_series", "amount_change", "descriptor_change", "unmatched"), cursor (cursorDate + cursorId), pageSize (default 20, max 50)
   - Fetch only unresolved items: `isNull(reviewQueueItems.resolvedAt)`
   - Join to get context: leftJoin recurringSeries (for series name/confidence), leftJoin recurringMasters (for suggested master name), leftJoin merchantEntities via recurringSeries.merchantEntityId (for merchant name)
   - Select: reviewQueueItems fields + seriesDetectedFrequency, seriesAvgAmount, seriesCurrency, merchantName, suggestedMasterName
   - Order by confidence DESC (highest confidence first — most actionable), then createdAt ASC
   - Cursor pagination on (confidence DESC, id ASC)
   - Response: `{ data: [...items], pagination: { hasMore, nextCursor } }`

2. Create `src/app/api/recurring/review-queue/[id]/resolve/route.ts` — POST:
   - Auth check + feature gate
   - Parse body with resolveReviewSchema
   - Verify review item exists, belongs to user, is unresolved (resolvedAt IS NULL)
   - Wrap ALL writes in single db.transaction() per DB pool constraint:

   **If resolution === "confirmed":**
   - Fetch the series linked to the review item
   - Create new recurringMaster: name from merchant entity name or "Unknown Recurring", recurringKind inferred from series (use inferRecurringKind from recurrence-linker.ts), derive amount/frequency/currency from series fields
   - Insert recurringMasterSeriesLinks (isPrimary: true)
   - Log recurringEvent type "created" on new master
   - Update reviewQueueItems: set resolvedAt, resolution="confirmed"

   **If resolution === "linked":**
   - Require targetMasterId in body, verify it exists and belongs to user
   - Insert recurringMasterSeriesLinks linking the review item's series to target master
   - Log recurringEvent type "linked" on target master with metadata { seriesId }
   - Update reviewQueueItems: set resolvedAt, resolution="linked"

   **If resolution === "ignored":**
   - Mark series as inactive: `db.update(recurringSeries).set({ isActive: false, updatedAt: new Date() })`
   - Update reviewQueueItems: set resolvedAt, resolution="ignored"

   **If resolution === "not_recurring":**
   - Fetch all transactions in the series via recurringSeriesTransactions
   - For each transaction, upsert userTransactionLabels with label="not_recurring"
   - Mark series as inactive
   - Update reviewQueueItems: set resolvedAt, resolution="not_recurring"

   - Always set resolutionNotes from body.notes
   - Log recurringEvent type "review_resolved" with metadata { resolution, reviewQueueItemId: id }
   - Return `{ success: true, resolution }`

3. Create `src/app/api/recurring/dashboard/route.ts` — GET:
   - Auth check
   - Single efficient query using SQL aggregates (NOT N+1):
   ```
   const [summary] = await db.execute(sql`
     SELECT
       (SELECT COUNT(*) FROM recurring_masters WHERE user_id = ${userId} AND status = 'active') as active_count,
       (SELECT COALESCE(SUM(expected_amount::numeric), 0) FROM recurring_masters WHERE user_id = ${userId} AND status = 'active') as monthly_total,
       (SELECT COUNT(*) FROM recurring_masters WHERE user_id = ${userId} AND status = 'active' AND next_expected_date IS NOT NULL AND next_expected_date <= NOW() + INTERVAL '7 days' AND next_expected_date >= NOW()) as upcoming_count,
       (SELECT COUNT(*) FROM review_queue_items WHERE user_id = ${userId} AND resolved_at IS NULL) as needs_review_count
   `);
   ```
   - Fetch upcoming payments (up to 10): `db.select({id, name, expectedAmount, currency, nextExpectedDate, recurringKind}).from(recurringMasters).where(and(eq(userId), eq(status, "active"), isNotNull(nextExpectedDate), gte(nextExpectedDate, now), lte(nextExpectedDate, now + 7 days))).orderBy(asc(nextExpectedDate)).limit(10)`
   - Fetch recent amount changes (up to 5): `db.select({...eventFields, masterName: recurringMasters.name}).from(recurringEvents).innerJoin(recurringMasters, eq(...)).where(and(eq(userId), eq(eventType, "amount_changed"))).orderBy(desc(createdAt)).limit(5)`
   - Response: `{ data: { activeCount, monthlyTotal, upcomingCount, needsReviewCount, upcomingPayments: [...], amountChanges: [...] } }`
   - Response is cache-friendly (no volatile data, UI can cache 60s)
