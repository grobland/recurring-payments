**Step 1: Install react-pdf**

Run `npm install react-pdf@^10.3.0`. Do NOT install `pdfjs-dist` separately — react-pdf bundles it. Version mismatch causes "API version does not match Worker version" errors.

No `next.config.ts` changes needed (confirmed: react-pdf v9.2+ eliminated this requirement).

**Step 2: Create `src/lib/hooks/use-pdf-url.ts`**

TanStack Query hook for lazy signed URL fetch. Key requirements:
- `enabled: !!statementId && enabled` — only fetches when modal is open
- `staleTime: 55 * 60 * 1000` (55 min, just under 1-hour signed URL expiry)
- `gcTime: 60 * 60 * 1000` (keep in cache for 1 hour)
- `retry: 1` — single retry on failure
- Query key: `["pdf-url", statementId]`
- Accept optional `download` boolean parameter. When true, append `?download=1` to the fetch URL. This fetches a signed URL with Content-Disposition: attachment.
- Return type: `{ url: string }`

**Step 3: Create `src/components/statements/pdf-viewer-inner.tsx`**

This is the file that imports from `react-pdf` and configures the worker. CRITICAL: Worker config MUST be in this same file as `<Document>`. "use client" directive required.

Requirements:
- Import `Document`, `Page`, `pdfjs` from `react-pdf`
- Import CSS: `react-pdf/dist/Page/AnnotationLayer.css` and `react-pdf/dist/Page/TextLayer.css`
- Worker config at module level: `pdfjs.GlobalWorkerOptions.workerSrc = new URL("pdfjs-dist/build/pdf.worker.min.mjs", import.meta.url).toString()`
- Props: `{ url: string; filename: string; onError?: () => void }`
- State: `numPages` (number, default 0), `pageNumber` (number, default 1)
- `onLoadSuccess` callback sets numPages and resets pageNumber to 1
- `onLoadError` calls `onError` prop
- `<Document>` with `loading` prop showing "Loading PDF..." text
- `<Page>` with `pageNumber`, `width={Math.min(760, typeof window !== "undefined" ? window.innerWidth - 80 : 760)}`, `renderTextLayer={true}`, `renderAnnotationLayer={true}`
- Page navigation: Show only when `numPages > 1`. Use shadcn `Button` (variant="outline", size="sm") for Previous/Next. `disabled` when at first/last page. Display "Page X of Y" between buttons. Use `ChevronLeft` and `ChevronRight` icons from lucide-react.
- Download button: An `<a>` element styled as a shadcn-like button. Use the `downloadUrl` prop (separate from `url`). If `downloadUrl` is not provided, fall back to `url`. Include `Download` icon from lucide-react. Text: "Download PDF". `download={filename}` attribute.
- Update props to: `{ url: string; downloadUrl?: string; filename: string; onError?: () => void }`

**Step 4: Create `src/components/statements/pdf-viewer-modal.tsx`**

Outer wrapper that dynamically imports the inner viewer. "use client" directive required.

Requirements:
- `dynamic(() => import("./pdf-viewer-inner").then((m) => ({ default: m.PdfViewerInner })), { ssr: false, loading: () => <div>Loading viewer...</div> })`
- Props: `{ statementId: string; filename: string; open: boolean; onClose: () => void }`
- State: `renderError` (boolean, default false). Reset to false when `open` changes to true (use useEffect).
- Call `usePdfUrl(statementId, open)` for the viewing URL (no download flag)
- Call `usePdfUrl` with download=true when user clicks download — OR simpler: pass a `downloadUrl` derived from appending `?download=1` to the base fetch. Actually, simplest approach: fetch two URLs is wasteful. Instead, construct the download URL by calling a second `usePdfUrl` with `download: true` only when the first URL succeeds. OR even simpler: just fetch ONE URL for viewing, and for download, construct the download endpoint URL directly as `/api/statements/${statementId}/pdf-url?download=1` and use that as an href. But that returns JSON, not the file. The cleanest approach: the modal fetches the viewing signed URL via `usePdfUrl`. For download, it fetches a SECOND signed URL with the download option. Use a separate call: `const { data: downloadData } = usePdfUrl(statementId, open && !!data?.url, true)` where the third param is `forDownload`.

Revised approach for simplicity: `usePdfUrl` returns `{ url: string; downloadUrl: string }` from a single API call. Update the API endpoint to return BOTH URLs in one response. This avoids double-fetching.

- Use shadcn `Dialog`, `DialogContent` (className="max-w-4xl max-h-[90vh] overflow-y-auto"), `DialogHeader`, `DialogTitle`
- `DialogTitle` shows filename
- Loading state: When `isLoading` from usePdfUrl, show "Preparing PDF..." centered text
- Error state: When `error` or no signedUrl after loading, show "Could not load PDF for viewing." message
- Render error state: When `signedUrl` exists but `renderError` is true, show "This PDF could not be rendered in-app." with a download fallback link `<a href={downloadUrl} download={filename}>Download PDF instead</a>`
- Main viewer: When `signedUrl` exists and no renderError, render `<PdfViewerInner url={signedUrl} downloadUrl={downloadUrl} filename={filename} onError={() => setRenderError(true)} />`

Export `PdfViewerModal` as named export.

**Step 1: Add download option to `generatePdfSignedUrl` in `src/lib/storage/pdf-storage.ts`**

Add an optional `options` parameter: `{ download?: string }`. When `download` is provided, pass `{ download: filename }` to `supabaseAdmin.storage.from(BUCKET).createSignedUrl(path, 3600, { download: filename })`. This makes Supabase set `Content-Disposition: attachment; filename=...` on the signed URL response. When `download` is not provided, call `createSignedUrl` without the download option (current behavior — for in-browser rendering).

Updated signature:
```typescript
export async function generatePdfSignedUrl(
  pdfStoragePath: string,
  options?: { download?: string }
): Promise<string | null>
```

**Step 2: Update `GET /api/statements/[id]/pdf-url/route.ts`**

Return BOTH a viewing URL and a download URL in the response. This means calling `generatePdfSignedUrl` twice:

1. `const viewUrl = await generatePdfSignedUrl(statement.pdfStoragePath)` — for in-browser rendering
2. `const downloadUrl = await generatePdfSignedUrl(statement.pdfStoragePath, { download: statement.originalFilename })` — for download with Content-Disposition

To get `originalFilename`, add it to the columns selected from the database query: `columns: { id: true, pdfStoragePath: true, originalFilename: true }`.

Return response: `{ url: viewUrl, downloadUrl: downloadUrl }`. If either URL generation fails, still return whichever succeeded (use `url: viewUrl ?? downloadUrl`, `downloadUrl: downloadUrl ?? viewUrl`). Both null = 500 error.

**Step 3: Update `usePdfUrl` return type**

The hook's return type should match: `Promise<{ url: string; downloadUrl: string }>`. Update the type assertion in the hook accordingly.
