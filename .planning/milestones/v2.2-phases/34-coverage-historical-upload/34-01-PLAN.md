  <name>Task 1: Create GET /api/vault/coverage endpoint</name>
  <files>src/app/api/vault/coverage/route.ts</files>
  <action>
Create the coverage API endpoint following the pattern in `/api/vault/timeline/route.ts`.

The endpoint must:
1. Authenticate via `auth()` and return 401 if no session
2. Compute the 12-month window: `months[0] = startOfMonth(subMonths(now, 11))` through `months[11] = startOfMonth(now)` -- exactly 12 months inclusive of current month
3. Run a single query fetching all statements for the user where `statementDate >= windowStart` (the same efficient pattern as timeline route -- no N+1)
4. Also query all distinct sourceTypes for the user (not just those in the window -- sources with older statements still need rows in the grid)
5. Group fetched statements into a Map keyed by `${sourceType}::${yyyy-MM}` format
6. For each source x month cell, determine state:
   - If no statements exist for that cell: `state: "missing"`, statementId: null, transactionCount: 0, statementDate: null
   - If at least one statement has `pdfStoragePath IS NOT NULL`: `state: "pdf"`, pick that statement for statementId
   - If statements exist but none has pdfStoragePath: `state: "data"`, pick the first statement for statementId
   - When multiple statements exist for a cell, prefer the one with a PDF (for state determination)
7. Compute `gapCount` as count of all "missing" cells across all sources
8. Return shape: `{ sources: CoverageSource[], gapCount: number, months: string[] }` where months is the 12 month labels as "yyyy-MM" strings ordered oldest to newest

Use `date-fns` functions: `startOfMonth`, `subMonths`, `format`. Use `gte`, `eq`, `and` from drizzle-orm. Import `statements` from schema. Never expose `pdfStoragePath` to client -- only use it to derive the cell state.

Important: statementDate may be null for some statements. Skip those when assigning to month cells (they have no month to map to). But their sourceType should still appear as a source row if they are the only statements for that source.
  </action>

Run `npx tsc --noEmit` to verify no type errors. Then start dev server and test:
- `curl http://localhost:3000/api/vault/coverage` without auth should return 401
- Verify the response shape matches the documented interface (sources array, gapCount number, months array of 12 strings)
  </verify>
  <done>

  </done>

  <files>src/app/api/vault/attach-pdf/route.ts</files>
  <action>
Create a lightweight endpoint for attaching a PDF to an existing statement record. This is used when a yellow cell (data exists, no PDF) is clicked -- the user wants to attach the original PDF without re-extracting transactions.

The endpoint must:
1. Authenticate via `auth()` and return 401 if no session
2. Parse FormData with fields: `file` (File) and `statementId` (string)
3. Validate: file must be present, must be PDF type, must be under 50MB; statementId must be present
4. Verify statement ownership: query `statements` table where `id = statementId AND userId = session.user.id`. Return 404 if not found (same ownership pattern as `/api/statements/[id]/pdf-url/route.ts`)
5. Call `uploadStatementPdf(file, session.user.id, statement.sourceType)` from `@/lib/storage/pdf-storage`
6. If storage upload returns null, return 500 with `{ error: "Storage upload failed" }`
7. Update the statement record: `SET pdfStoragePath = storageResult.path WHERE id = statementId`
8. Return `{ success: true, pdfStored: true }`

Do NOT run any transaction extraction or processing. This endpoint is purely for storage attachment.

Follow the existing pattern in `/api/batch/upload/route.ts` for FormData parsing and error handling. Import `isUserActive` from `@/lib/auth/helpers` and check trial/billing status (return 403 if expired, same as batch upload).
  </action>
  <verify>
Run `npx tsc --noEmit` to verify no type errors. The endpoint should compile cleanly with proper types for all drizzle operations.
  </verify>
  <done>
POST /api/vault/attach-pdf accepts FormData with file + statementId, uploads PDF to Supabase Storage, updates pdfStoragePath on the statement, and returns success without triggering transaction extraction. Ownership verification prevents accessing other users' statements.
  </done>
