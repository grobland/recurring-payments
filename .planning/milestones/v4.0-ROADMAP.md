## Phase Details

### Phase 47: Schema & Domain Model
**Goal**: Create all database tables, enums, and indexes for the three-layer recurring payment model
**Depends on**: Phase 46 (v3.2 complete)
**Requirements**: SCHEMA-01 through SCHEMA-17
**Success Criteria** (what must be TRUE):
  1. All new enums created: recurring_kind (8 values), recurring_status (5 values), amount_type (2 values)
  2. All new tables created with UUID PKs: merchant_entities, merchant_aliases, raw_statement_lines, recurring_series, recurring_series_transactions, recurring_masters, recurring_master_series_links, user_transaction_labels, review_queue_items, recurring_events
  3. Transactions table refactored: normalized_description added, source_hash added with uniqueness constraint
  4. Trigram indexes created on description columns and alias tables
  5. All foreign keys reference correct tables with appropriate cascade behavior
  6. Migration runs cleanly on existing database without data loss
**Plans**: 2 plans
Plans:
- [x] 47-01-PLAN.md — Define enums, tables, columns, and indexes in schema.ts
- [x] 47-02-PLAN.md — Wire relations, add type exports, generate migration with trigram indexes

### Phase 48: Ingestion Pipeline & Merchant Resolution
**Goal**: Build the complete PDF-to-normalized-transactions pipeline with merchant entity resolution
**Depends on**: Phase 47
**Requirements**: INGEST-01 through INGEST-10, MERCH-01 through MERCH-05, SEC-01 through SEC-04
**Success Criteria** (what must be TRUE):
  1. PDF upload validates file type and size, stores temporarily
  2. Parser abstraction interface exists with generic implementation using existing OpenAI extraction
  3. Raw extracted lines stored in raw_statement_lines preserving original text
  4. Transactions normalized with normalized_description and source_hash
  5. Re-upload of same statement detects duplicates via source_hash
  6. Merchant resolution matches descriptors to entities via aliases with fuzzy fallback
  7. Transfers and refunds filtered where detectable
  8. Pipeline runs all 10 steps in order for each uploaded PDF
**Plans**: 3 plans
Plans:
- [x] 48-01-PLAN.md — Pure utility functions: merchant normalization, transfer detection, source hash, parser abstraction
- [x] 48-02-PLAN.md — Merchant resolver service with exact/fuzzy/create-new resolution
- [x] 48-03-PLAN.md — Wire pipeline into batch/process route with full verification

### Phase 49: Recurrence Detection & Auto-Linking
**Goal**: Detect recurring patterns in transactions and auto-link to existing masters with confidence scoring
**Depends on**: Phase 48
**Requirements**: DETECT-01 through DETECT-10, LINK-01 through LINK-09
**Success Criteria** (what must be TRUE):
  1. Rule A (alias hit), Rule B (fixed monthly), Rule C (variable monthly), Rule D (annual) all working
  2. Weekly, quarterly, and custom cadence detection supported
  3. Fixed vs variable amount_type distinguished automatically
  4. Next payment date predicted from detected cadence
  5. Auto-link at >= 0.85 confidence, review queue at 0.60-0.84, unmatched below 0.60
  6. Descriptor changes, amount changes, and account migration handled gracefully
  7. User manual decisions override automation and influence future matching
  8. Detection decisions logged in recurring_events
**Plans**: 3 plans
Plans:
- [x] 49-01-PLAN.md — Core detection engine with rules A-E, series creation, amount classification, next-date prediction
- [x] 49-02-PLAN.md — Auto-linking engine with confidence thresholds, review queue, audit trail
- [x] 49-03-PLAN.md — Orchestrator wiring detector + linker into batch/process pipeline
