## E2E User Flows

### Flow 1: New User Subscription
**Status:** COMPLETE

1. User signs up -> lands on 14-day trial (billingStatus: "trial")
2. Navigates to /settings/billing -> sees 3 tier cards with pricing from /api/billing/prices
3. Clicks "Subscribe" on desired tier -> handleCheckout sends tier/interval/currency
4. POST /api/billing/create-checkout -> getPriceIdForCheckout -> Stripe Checkout session
5. allow_promotion_codes: true -> user can enter voucher code
6. Completes payment -> checkout.session.completed webhook fires
7. handleCheckoutCompleted updates user: billingStatus "active", stripePriceId, stripeSubscriptionId
8. Features unlocked based on new tier

### Flow 2: Feature Access Control
**Status:** COMPLETE

1. Trial/unpaid user navigates to analytics page -> FeatureGate checks BASIC_ANALYTICS (primary tier)
2. Null tier treated as primary -> trial users pass primary gates
3. LockedNavItem grays out "Spending Monitor" (enhanced tier) in sidebar for primary/trial users
4. Import API enforces requireFeature(PDF_IMPORTS) -> 403 if user lacks tier access
5. Upgrade modal shows when clicking locked FeatureGate content -> links to /pricing and /settings/billing

### Flow 3: Payment Failed Email
**Status:** COMPLETE

1. Payment fails -> invoice.payment_failed webhook fires
2. handlePaymentFailed updates billingStatus to "past_due"
3. First-attempt check prevents spam emails
4. Stripe billing portal session created with flow_data: payment_method_update
5. return_url: /settings/billing (corrected by Phase 30)
6. renderPaymentFailedEmail sent via Resend with portal link
7. User clicks email -> Stripe portal -> updates payment -> returns to /settings/billing

### Flow 4: Portal Management
**Status:** COMPLETE

1. User goes to /settings/billing -> sees current tier highlighted with "Current Plan" badge
2. Clicks "Manage Billing" -> POST /api/billing/portal -> Stripe portal session
3. Portal shows app branding (logo, colors)
4. User can switch tiers with prorated billing
5. customer.subscription.updated webhook fires -> stripePriceId updated
6. Feature access recalculated on next getUserTier() call (no stale cache)
7. Returns to /settings/billing

### Flow 5: Voucher Redemption
**Status:** COMPLETE

1. Admin creates coupon in Stripe Dashboard (per docs/admin/voucher-workflow.md)
2. Admin creates promotion code from coupon with max_redemptions
3. User enters code during Stripe Checkout (allow_promotion_codes: true)
4. Stripe validates and applies discount to subscription

### Flow 6: Admin Trial Extension
**Status:** COMPLETE

1. Admin user signs in (bootstrapped via seed-admin.ts with ADMIN_EMAIL)
2. Sidebar shows Admin section with "Trial Extensions" and "Webhooks" links (isAdmin check)
3. Admin navigates to /admin/trial-extensions -> layout guard checks role
4. Page shows trial users dropdown and extension history
5. Admin submits extension -> POST /api/admin/trial-extensions (admin role 403 guard)
6. API validates, calculates cumulative new trial end date, updates user, creates audit record
7. Non-admin users redirected to /dashboard (layout) or get 403 (API)
