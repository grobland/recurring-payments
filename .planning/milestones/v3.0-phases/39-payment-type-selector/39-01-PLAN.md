## Goal

Install the two new dependencies (nuqs, shadcn toggle-group), wire NuqsAdapter into the provider tree, extend the transaction type system with a `paymentType` field, update the `useTransactions` hook to pass the new filter param, and implement the server-side paymentType filter logic in the transactions API route (including the recurringPatterns LEFT JOIN for recurring/one-time classification).

After this plan, the API correctly filters transactions by payment type when `?paymentType=recurring|subscriptions|one-time` is passed, and the nuqs infrastructure is ready for UI consumption.

## Locked Decisions (from CONTEXT.md — non-negotiable)

- nuqs@^2.8.8 (pre-decided in STATE.md)
- Four payment types: all, recurring, subscriptions, one-time
- "All" is the default (no URL param = All)
- Classification: Recurring = merchant in recurringPatterns OR tagStatus IN ('potential_subscription', 'converted'); Subscriptions = convertedToSubscriptionId IS NOT NULL OR tagStatus = 'converted'; One-time = NOT recurring AND NOT subscription
- Filter must be server-side (cursor-based pagination — client-side filtering breaks pagination)

## Claude's Discretion

- Exact Drizzle query structure for the recurringPatterns LEFT JOIN (subquery vs direct join)
- Whether to refactor the sourceType two-branch query into a single query builder
- Error handling for invalid paymentType values

<task id="39-01-T3" title="Implement paymentType filter in transactions API route" autonomous="true">

### What
Extend `GET /api/transactions` to parse the `paymentType` query param and apply the correct DB conditions, including a LEFT JOIN on `recurringPatterns` for recurring/one-time classification.

### Why
The payment type filter MUST be server-side because the transaction list uses cursor-based keyset pagination. Client-side filtering would break page sizes and infinite scroll.

### How

1. **Parse the paymentType param** in `src/app/api/transactions/route.ts`:
   ```typescript
   const paymentType = searchParams.get("paymentType");
   ```
   Add this after the existing `accountId` param parsing (around line 26).

2. **Import recurringPatterns** from the schema:
   ```typescript
   import { transactions, statements, transactionTags, tags, subscriptions, categories, recurringPatterns } from "@/lib/db/schema";
   import { and, eq, or, lt, gte, lte, ilike, desc, inArray, sql, isNotNull, isNull, notInArray } from "drizzle-orm";
   ```

3. **Refactor the two-branch query into a single query builder:**
   The current code has two branches: one for `sourceType` and one without. Replace with a unified approach where the `sourceType` condition is added to the `conditions[]` array (which is already done for `accountId`). This eliminates the if/else branching:

   ```typescript
   // Move sourceType into conditions array (like accountId already is)
   if (sourceType) {
     conditions.push(eq(statements.sourceType, sourceType));
   }
   ```

   Then a single query:
   ```typescript
   results = await db
     .select(selectFields)
     .from(transactions)
     .leftJoin(statements, eq(transactions.statementId, statements.id))
     .leftJoin(subscriptions, eq(transactions.convertedToSubscriptionId, subscriptions.id))
     .leftJoin(categories, eq(subscriptions.categoryId, categories.id))
     .where(and(...conditions))
     .orderBy(desc(transactions.transactionDate), desc(transactions.id))
     .limit(PAGE_SIZE + 1);
   ```

4. **Add paymentType filter conditions:**
   Apply conditions BEFORE the query execution, based on the `paymentType` param value:

   **For `subscriptions`:**
   ```typescript
   if (paymentType === 'subscriptions') {
     conditions.push(
       or(
         isNotNull(transactions.convertedToSubscriptionId),
         eq(transactions.tagStatus, 'converted')
       )!
     );
   }
   ```

   **For `recurring`:**
   Use a subquery to check if the merchant exists in `recurringPatterns` for this user, OR the transaction has a subscription-related tagStatus:
   ```typescript
   if (paymentType === 'recurring') {
     const recurringMerchantSubquery = db
       .select({ merchantName: recurringPatterns.merchantName })
       .from(recurringPatterns)
       .where(eq(recurringPatterns.userId, session.user.id));

     conditions.push(
       or(
         inArray(sql`LOWER(${transactions.merchantName})`,
           sql`(SELECT LOWER(${recurringPatterns.merchantName}) FROM ${recurringPatterns} WHERE ${recurringPatterns.userId} = ${session.user.id})`
         ),
         eq(transactions.tagStatus, 'potential_subscription'),
         eq(transactions.tagStatus, 'converted'),
         isNotNull(transactions.convertedToSubscriptionId)
       )!
     );
   }
   ```

   **For `one-time`:**
   The inverse of recurring — merchant NOT in recurringPatterns AND no subscription-related status:
   ```typescript
   if (paymentType === 'one-time') {
     conditions.push(
       sql`LOWER(${transactions.merchantName}) NOT IN (SELECT LOWER(${recurringPatterns.merchantName}) FROM ${recurringPatterns} WHERE ${recurringPatterns.userId} = ${session.user.id})`
     );
     conditions.push(
       or(
         eq(transactions.tagStatus, 'unreviewed'),
         eq(transactions.tagStatus, 'not_subscription')
       )!
     );
     conditions.push(isNull(transactions.convertedToSubscriptionId));
   }
   ```

   **For `all` (or absent):** No additional conditions — existing behavior.

5. **Validate paymentType:** If the param is provided but not one of the valid values, ignore it (treat as `all`):
   ```typescript
   const validPaymentTypes = ['recurring', 'subscriptions', 'one-time'];
   const effectivePaymentType = paymentType && validPaymentTypes.includes(paymentType)
     ? paymentType
     : null;
   ```
   Then use `effectivePaymentType` in the condition blocks.

### Implementation Notes

- The `sql` template literal for the subquery is the cleanest approach because Drizzle's `inArray` does not directly support subqueries on different columns. Using raw SQL for the subquery avoids complex Drizzle API gymnastics.
- The `LOWER()` comparison ensures case-insensitive merchant matching between `transactions.merchantName` and `recurringPatterns.merchantName`.
- The one-time filter uses `tagStatus IN ('unreviewed', 'not_subscription')` — this explicitly EXCLUDES `potential_subscription` and `converted` from one-time results. A transaction that was dismissed (`not_subscription`) is correctly classified as one-time.

- `GET /api/transactions?paymentType=subscriptions` returns only transactions with `convertedToSubscriptionId IS NOT NULL` or `tagStatus = 'converted'`
- `GET /api/transactions?paymentType=recurring` returns transactions whose merchant appears in `recurringPatterns` for the user OR has `tagStatus` of `potential_subscription`/`converted` or a `convertedToSubscriptionId`
- `GET /api/transactions?paymentType=one-time` returns transactions NOT matching recurring criteria
