## Goal

Build the PaymentTypeSelector segmented control component, wire it into TransactionBrowser using nuqs `useQueryState` for URL-persisted state, add an inline subscription checkbox on recurring transaction rows, and implement contextual empty state messages per payment type. After this plan, users can filter transactions by All/Recurring/Subscriptions/One-time with the selection persisted in the URL and combined with all existing filters.

## Locked Decisions (from CONTEXT.md — non-negotiable)

- Segmented control (connected pill group, iOS-style) — active segment has filled background, others ghost/outline
- Positioned ABOVE all existing filters (search, date range, tag status)
- No transaction counts on segments — clean labels only
- Four segments in order: All, Recurring, Subscriptions, One-time
- Full words, no abbreviations
- Default: "All" on first page load (no URL param = All)
- Empty filter type: contextual empty state message (e.g., "No one-time payments found")
- Segments never disabled — all four always clickable
- No onboarding tooltip
- Inline subscription checkbox visible directly on each recurring transaction row
- Auto-suggested but unconfirmed subscriptions have a subtle visual indicator
- Confirmed subscriptions look different from suggested ones

## Claude's Discretion

- Segmented control width (fit-content vs full-width)
- Loading/transition states when switching segments
- Suggested indicator exact visual treatment (dot, badge, outline style)
- Error state handling
- Subscription checkbox exact position within the row

## Tasks

<task id="39-02-T1" title="Create PaymentTypeSelector component" autonomous="true">

### What
Create a new `PaymentTypeSelector` component that renders a segmented control (iOS-style connected pill group) with four segments: All, Recurring, Subscriptions, One-time. Uses shadcn `ToggleGroup` under the hood.

### Why
This is the primary UI control for FILTER-01. The user needs a visible, clickable segmented control to switch between payment types. It must be a standalone component so it can be placed above the existing filters in TransactionBrowser.

### How

1. **Create `src/components/transactions/payment-type-selector.tsx`:**

   ```tsx
   "use client";

   import { ToggleGroup, ToggleGroupItem } from "@/components/ui/toggle-group";
   import type { PaymentType } from "@/types/transaction";
   import { PAYMENT_TYPES } from "@/types/transaction";

   interface PaymentTypeSelectorProps {
     value: PaymentType;
     onChange: (value: PaymentType) => void;
   }

   const LABELS: Record<PaymentType, string> = {
     all: "All",
     recurring: "Recurring",
     subscriptions: "Subscriptions",
     "one-time": "One-time",
   };

   export function PaymentTypeSelector({ value, onChange }: PaymentTypeSelectorProps) {
     return (
       <ToggleGroup
         type="single"
         value={value}
         onValueChange={(v) => {
           // Prevent deselection — Radix sends "" when clicking the active item
           if (v) onChange(v as PaymentType);
         }}
         className="w-fit"
       >
         {PAYMENT_TYPES.map((type) => (
           <ToggleGroupItem
             key={type}
             value={type}
             aria-label={`Show ${LABELS[type].toLowerCase()} transactions`}
             className="px-4 text-sm"
           >
             {LABELS[type]}
           </ToggleGroupItem>
         ))}
       </ToggleGroup>
     );
   }
   ```

2. **Style the segmented control as iOS-style connected pills:**
   The shadcn toggle-group already provides a connected group appearance. To achieve the iOS-style filled-active / ghost-inactive look, apply styling via Tailwind classes on the ToggleGroupItem:
   - Active: filled background (the default `data-[state=on]` variant from shadcn's toggle-group handles this)
   - Inactive: ghost/outline appearance (default `data-[state=off]` behavior)

   If the default shadcn styling does not achieve the connected-pill look, add custom classes:
   ```tsx
   <ToggleGroup
     type="single"
     value={value}
     onValueChange={(v) => { if (v) onChange(v as PaymentType); }}
     className="w-fit border rounded-lg p-1 bg-muted/50"
   >
     {PAYMENT_TYPES.map((type) => (
       <ToggleGroupItem
         key={type}
         value={type}
         className="px-4 text-sm rounded-md data-[state=on]:bg-background data-[state=on]:shadow-sm"
       >
         {LABELS[type]}
       </ToggleGroupItem>
     ))}
   </ToggleGroup>
   ```

   Use your discretion on exact padding, border-radius, and shadow to get a polished connected-pill appearance that fits the existing app aesthetic.

3. **Export from index.ts:**
   Add to `src/components/transactions/index.ts`:
   ```typescript
   export { PaymentTypeSelector } from "./payment-type-selector";
   ```

### Verification
- Component renders four segments: All, Recurring, Subscriptions, One-time
- Clicking a segment calls `onChange` with the correct value
- Clicking the already-active segment does NOT deselect it (value stays the same)
- Component is keyboard-navigable (arrow keys between segments)
- No transaction counts appear on segments (clean labels only)

</task>

<task id="39-02-T2" title="Wire nuqs and PaymentTypeSelector into TransactionBrowser" autonomous="true">

### What
Replace the implicit "all transactions" default in `TransactionBrowser` with a nuqs-powered `paymentType` URL state. Add the `PaymentTypeSelector` above existing filters. Feed `paymentType` into the `debouncedFilters` that drive `useTransactions`.

### Why
This is the core integration for FILTER-01, FILTER-02, and FILTER-03. The `paymentType` state lives in the URL via nuqs, ensuring it persists across page loads (FILTER-02). It feeds into `useTransactions` which passes it to the API, combining with all existing filters (FILTER-03). The segmented control renders above existing filters for user visibility (FILTER-01).

### How

1. **Update `src/components/transactions/transaction-browser.tsx`:**

   Add imports:
   ```tsx
   import { useQueryState, parseAsStringLiteral } from 'nuqs';
   import { PaymentTypeSelector } from "./payment-type-selector";
   import { PAYMENT_TYPES } from "@/types/transaction";
   import type { PaymentType } from "@/types/transaction";
   ```

   Add the nuqs state hook inside `TransactionBrowser`, before the existing `useState` for filters:
   ```tsx
   const [paymentType, setPaymentType] = useQueryState(
     'paymentType',
     parseAsStringLiteral(PAYMENT_TYPES).withDefault('all')
   );
   ```

   Modify the `debouncedFilters` useMemo to include `paymentType`:
   ```tsx
   const debouncedFilters = useMemo(
     () => ({
       ...filters,
       search: debouncedSearch,
       paymentType: paymentType === 'all' ? undefined : paymentType,
     }),
     [filters, debouncedSearch, paymentType]
   );
   ```

   Update the `hasActiveFilters` check to include `paymentType`:
   ```tsx
   const hasActiveFilters =
     filters.search ||
     filters.sourceType ||
     filters.tagStatus ||
     filters.dateFrom ||
     filters.dateTo ||
     paymentType !== 'all';
   ```

2. **Add PaymentTypeSelector above TransactionFilters in all render paths:**
   In the JSX (loading state, error state, empty state, and main content), add the `PaymentTypeSelector` above the `TransactionFilters` component. Wrap them in a container:

   ```tsx
   <div className="flex flex-col gap-3 mb-0">
     <PaymentTypeSelector value={paymentType} onChange={setPaymentType} />
     <TransactionFilters
       filters={filters}
       onFiltersChange={setFilters}
       sourceTypes={uniqueSourceTypes}
     />
   </div>
   ```

   Apply this pattern in all four render branches:
   - Loading state (line ~156-168)
   - Error state (line ~173-194)
   - Empty state (line ~199-231)
   - Main content (line ~236-274)

   **Important:** Remove the duplicate `mb-4` from `TransactionFilters` parent since the wrapper handles spacing. The `TransactionFilters` component has its own sticky positioning and `pb-4 mb-4` — these may need adjustment so that the PaymentTypeSelector scrolls with the page while the filter bar remains sticky. Use your discretion: either make both sticky (PaymentTypeSelector + filters in one sticky container) or keep only the filter bar sticky.

3. **Update the "Clear filters" button in empty state** to also reset `paymentType`:
   ```tsx
   <Button
     onClick={() => {
       setFilters({});
       setPaymentType('all');
     }}
     variant="outline"
     className="gap-2"
   >
     Clear filters
   </Button>
   ```

4. **Contextual empty state messages per payment type:**
   Update the empty state messaging to reflect the selected payment type:
   ```tsx
   const emptyMessages: Record<PaymentType, string> = {
     all: "No transactions found",
     recurring: "No recurring payments found",
     subscriptions: "No subscriptions found",
     "one-time": "No one-time payments found",
   };

   // In the empty state:
   <h3 className="text-lg font-semibold">
     {hasActiveFilters
       ? emptyMessages[paymentType]
       : "No transactions found"}
   </h3>
   ```

5. **Pass paymentType to child components** for the inline subscription checkbox (Task 39-02-T3):
   Pass `paymentType` as a prop to `TransactionTable` and `TransactionCardList`:
   ```tsx
   <TransactionTable
     transactions={allTransactions}
     paymentType={paymentType}
     // ...existing props
   />
   ```
   And:
   ```tsx
   <TransactionCardList
     transactions={allTransactions}
     paymentType={paymentType}
     // ...existing props
   />
   ```

### Verification
- PaymentTypeSelector appears above the existing filter bar
- Clicking "Recurring" updates URL to `?paymentType=recurring`
- Clicking "All" removes the `paymentType` param from the URL (clean URL)
- The transaction list updates to show only filtered results

<task id="39-02-T3" title="Add inline subscription checkbox on recurring transaction rows" autonomous="true">

### What
Add an inline checkbox to transaction rows that appears when viewing the "recurring" payment type. The checkbox shows subscription status: confirmed (checked), suggested (checked with visual indicator), or plain recurring (unchecked). Toggling the checkbox updates the transaction's tagStatus.

### Why
This fulfills the CONTEXT.md requirement for inline subscription confirmation. Users browsing recurring payments can quickly confirm or dismiss subscription suggestions without navigating to a detail view.

### How

1. **Update `TransactionTable` and `TransactionCardList`** to accept and pass through the `paymentType` prop:

   In `src/components/transactions/transaction-table.tsx`:
   - Add `paymentType?: PaymentType` to the props interface (import `PaymentType` from types)
   - Pass `paymentType` down to each `TransactionRow`

   In `src/components/transactions/transaction-card-list.tsx`:
   - Add `paymentType?: PaymentType` to the props interface
   - Pass `paymentType` down to each `TransactionCard`

2. **Update `TransactionRow`** (`src/components/transactions/transaction-row.tsx`):

   Add `paymentType` to the props interface:
   ```tsx
   interface TransactionRowProps {
     transaction: TransactionWithSource;
     style: CSSProperties;
     isSelected: boolean;
     onToggle: () => void;
     paymentType?: PaymentType;
   }
   ```

   Add subscription status logic:
   ```tsx
   const isSubscriptionConfirmed =
     transaction.convertedToSubscriptionId !== null ||
     transaction.tagStatus === 'converted';

   const isSubscriptionSuggested =
     transaction.tagStatus === 'potential_subscription';

   const showSubscriptionCheckbox = paymentType === 'recurring';
   ```

   Add a mutation hook for toggling subscription status. Use the existing tag system or create a simple PATCH call:
   ```tsx
   const handleSubscriptionToggle = async (checked: boolean) => {
     // If checking: set tagStatus to 'converted' (confirming subscription)
     // If unchecking: revert to 'potential_subscription' (if was suggested) or 'unreviewed'
     // Use existing convert transaction hook or PATCH /api/transactions/:id
   };
   ```

   For the actual toggle mutation, reuse the existing `useConvertTransaction` hook when checking (it already converts a transaction to a subscription). For unchecking, use a simple fetch to `PATCH /api/transactions/:id` to update tagStatus back to `potential_subscription` or `unreviewed`.

   **However, the simplest approach that aligns with existing patterns:** Show the checkbox as read-only state indicator in the row. Checking it triggers the existing "Convert" flow (same as the Convert button). Unchecking is a less common action — for v1, the checkbox can be one-way (check to confirm, use existing UI to unconvert if needed).

   Add the checkbox column in the row JSX, positioned after the Tag Status column and before the Actions column:
   ```tsx
   {showSubscriptionCheckbox && (
     <td className="w-[50px] px-2 py-3 flex items-center justify-center">
       <div className="relative">
         <Checkbox
           checked={isSubscriptionConfirmed || isSubscriptionSuggested}
           onCheckedChange={(checked) => {
             if (checked && !isSubscriptionConfirmed) {
               convertTransaction.mutate({
                 transactionId: transaction.id,
                 merchantName: transaction.merchantName,
               });
             }
           }}
           disabled={isSubscriptionConfirmed || convertTransaction.isPending}
           aria-label={`Mark ${transaction.merchantName} as subscription`}
         />
         {/* Suggested indicator: subtle dot/badge for auto-suggested but unconfirmed */}
         {isSubscriptionSuggested && !isSubscriptionConfirmed && (
           <span
             className="absolute -top-1 -right-1 h-2 w-2 rounded-full bg-amber-400"
             title="Suggested subscription"
           />
         )}
       </div>
     </td>
   )}
   ```

   The visual distinction:
   - **Confirmed subscription:** Checkbox checked, disabled (already converted) — standard checkbox appearance
   - **Suggested subscription:** Checkbox checked (pre-checked), enabled, with amber dot indicator — user can see it's a suggestion
   - **Plain recurring:** Checkbox unchecked, enabled — user can check to convert

3. **Update `TransactionCard`** similarly for mobile view:
   Add the same subscription checkbox logic but adapted for the card layout. Position it at the end of the card, visible only when `paymentType === 'recurring'`.

4. **Adjust table header** in `TransactionTable`:
   When `paymentType === 'recurring'`, add a "Sub?" column header for the subscription checkbox column. When not recurring, hide the column.

### Implementation Notes

- The subscription checkbox column is conditionally shown — it only appears when `paymentType === 'recurring'`. This avoids clutter in other views.
- The amber dot indicator for suggested subscriptions provides the "subtle visual indicator" required by CONTEXT.md. Confirmed subscriptions have no dot (clean checked state).
- The checkbox reuses the existing `useConvertTransaction` hook for the confirm action, maintaining consistency with the existing conversion flow.
- Width adjustment: When the subscription checkbox column is shown, the other columns may need slight width reductions to accommodate it. Use your discretion on exact widths.

### Verification
- When `paymentType === 'recurring'`: each transaction row shows a subscription checkbox
- Confirmed subscriptions (convertedToSubscriptionId set) show checked + disabled checkbox
- Suggested subscriptions (tagStatus === 'potential_subscription') show checked checkbox with amber dot
- Plain recurring transactions show unchecked checkbox
- Checking an unchecked checkbox triggers the convert-to-subscription flow
- When `paymentType !== 'recurring'`: no subscription checkbox column appears
- Table column widths adjust properly when the checkbox column is shown/hidden
- Mobile card view also shows the checkbox when in recurring mode
