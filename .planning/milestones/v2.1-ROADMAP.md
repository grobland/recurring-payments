# Milestone v2.1: Billing & Monetization

**Status:** In Progress (Gap Closure)
**Phases:** 24-30
**Total Plans:** 16 (+ gap closure plans TBD)

## Overview

Enable paid subscriptions with three-tier feature access (Primary, Enhanced, Advanced), reliable Stripe integration, and promotional capabilities. This is an architectural expansion of the existing billing foundation to support multi-tier monetization.

## Phases

### Phase 24: Webhook Infrastructure Hardening

**Goal**: System reliably processes Stripe webhook events without duplicates or missed events
**Depends on**: Phase 23
**Requirements**: HOOK-01, HOOK-02
**Success Criteria** (what must be TRUE):
  1. Webhook events are processed exactly once, even during Stripe retries
  2. Processed event IDs are tracked in a database table
  3. User receives email when payment fails
  4. Webhook handler completes within Stripe timeout limits
**Plans**: 3 plans

Plans:
- [x] 24-01-PLAN.md - Schema + Idempotency (Wave 1)
- [x] 24-02-PLAN.md - Email, Health, Cleanup (Wave 2)
- [x] 24-03-PLAN.md - Admin Webhook Logs UI (Wave 3)

### Phase 25: Multi-Tier Product Setup

**Goal**: Users can subscribe to one of three tiers with correct pricing and tier-aware access
**Depends on**: Phase 24
**Requirements**: TIER-01, TIER-02, TIER-03
**Success Criteria** (what must be TRUE):
  1. Three products exist in Stripe (Primary, Enhanced, Advanced)
  2. Each product has monthly and annual prices configured
  3. System maps price ID to tier for any subscription
  4. Original pricing is preserved when tier prices change (grandfathering)
  5. User's tier is determined by their stripePriceId, not a separate column
**Plans**: 5 plans

Plans:
- [x] 25-01-PLAN.md - Database schema for price-to-tier mapping (Wave 1)
- [x] 25-02-PLAN.md - Tier derivation logic and checkout update (Wave 2)
- [x] 25-03-PLAN.md - Stripe products setup and price seeding (Wave 2)
- [x] 25-04-PLAN.md - Multi-tier billing UI (Gap closure, Wave 1)
- [x] 25-05-PLAN.md - Multi-tier public pricing page (Gap closure, Wave 2)

### Phase 26: Feature Gating Infrastructure

**Goal**: System can gate features by tier and show upgrade prompts for locked features
**Depends on**: Phase 25
**Requirements**: GATE-01, GATE-02, GATE-03
**Success Criteria** (what must be TRUE):
  1. Server-side hasFeature() utility checks user access to specific features
  2. API routes can enforce tier requirements with requireFeature() middleware
  3. Each tier has a defined set of accessible features in configuration
  4. Client-side FeatureGate component renders upgrade prompt for locked features
  5. Trial users have access to Primary tier features (standard tier for trial)
**Plans**: 2 plans

Plans:
- [x] 26-01-PLAN.md - Feature configuration and server utilities (Wave 1)
- [x] 26-02-PLAN.md - Client-side FeatureGate component (Wave 2)

### Phase 27: Pricing & Portal UI

**Goal**: Users can view pricing, access customer portal, and manage their subscription
**Depends on**: Phase 26
**Requirements**: PORTAL-01, PORTAL-02, PORTAL-03
**Success Criteria** (what must be TRUE):
  1. User can access Stripe customer portal from billing settings
  2. User can switch between tiers with prorated billing in portal
  3. Customer portal displays app branding (logo and colors)
  4. Pricing page shows all three tiers with feature comparison
  5. Current tier is highlighted in billing settings
**Plans**: 3 plans

Plans:
- [x] 27-01-PLAN.md - Feature comparison table for pricing page (Wave 1)
- [x] 27-02-PLAN.md - Current tier highlighting in billing settings (Wave 1)
- [x] 27-03-PLAN.md - Stripe portal configuration (Wave 2, human action)

### Phase 28: Voucher System

**Goal**: Admin can create vouchers and users can redeem them for free months
**Depends on**: Phase 25 (products must exist)
**Requirements**: VCHR-01, VCHR-02, VCHR-03
**Success Criteria** (what must be TRUE):
  1. Admin can create voucher codes in Stripe with redemption limits
  2. User can apply voucher code during checkout flow
  3. Admin can create trial extension vouchers for engaged users
  4. Vouchers have max_redemptions and first_time_transaction restrictions
  5. Checkout page shows promo code input field (already enabled via allow_promotion_codes)
**Plans**: 3 plans

Plans:
- [x] 28-01-PLAN.md - Trial extension schema + API endpoint (Wave 1)
- [x] 28-02-PLAN.md - Admin trial extensions UI (Wave 2)
- [x] 28-03-PLAN.md - Voucher verification and documentation (Wave 3)

### Phase 29: Apply Feature Gating (Gap Closure)

**Goal**: Feature gating infrastructure is actually applied to features in the application
**Depends on**: Phase 26
**Requirements**: GATE-03
**Gap Closure**: Closes gaps from v2.1 audit
**Success Criteria** (what must be TRUE):
  1. At least one feature/page is wrapped with FeatureGate component
  2. At least one API route enforces tier via requireFeature()
  3. Users without required tier see upgrade modal when accessing locked content
  4. LockedNavItem is used for at least one navigation item
  5. Trial users can access Primary tier features (unchanged from Phase 26)
**Plans**: 1 plan

Plans:
- [x] 29-01-PLAN.md - Wire feature gating to import API, analytics page, and sidebar nav

### Phase 30: Fix URLs & Admin Security (Gap Closure)

**Goal**: Payment failed flow works end-to-end and admin routes are secured
**Depends on**: Phase 24, Phase 28
**Requirements**: HOOK-02
**Gap Closure**: Closes gaps from v2.1 audit
**Success Criteria** (what must be TRUE):

  2. Admin routes check for admin role before allowing access
  3. Admin navigation links visible to admin users only
  4. Dead imports removed (TIER_LEVELS, getTierDisplayName, getTierFeatures)
  5. Navigation link to /pricing exists from marketing header or dashboard
**Plans**: 2 plans
