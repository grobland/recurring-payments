<task type="checkpoint:human-verify" gate="blocking">
  <name>Checkpoint: Review generated SQL for FK bug #4147 corruption</name>
  <files>src/lib/db/migrations/0011_*.sql</files>
  <action>
PAUSE — human must review the generated SQL printed above before migration is applied.

Drizzle bug #4147 can produce a malformed FK reference when adding a FK column to an existing table. Specifically, the `account_id` column added to `statements` may have a garbled REFERENCES clause. A corrupt migration applied to Supabase cannot be safely rolled back.

The human must read the SQL and confirm it is correct before Claude proceeds to run db:migrate.
  </action>
  <verify>
Read the generated SQL file above and verify ALL of the following:

1. `CREATE TYPE "public"."account_type" AS ENUM('bank_debit', 'credit_card', 'loan');` is present.

2. `CREATE TABLE "financial_accounts" (...)` is present with:
   - `"id" uuid PRIMARY KEY DEFAULT gen_random_uuid() NOT NULL`
   - `"user_id" uuid NOT NULL`
   - `"account_type" "account_type" NOT NULL`
   - Nullable type-specific columns: `credit_limit`, `statement_closing_day`, `original_balance`, `interest_rate`, `loan_term_months`

3. The FK for `financial_accounts.user_id` references `users(id)` with `ON DELETE cascade`.

4. **CRITICAL — check for Drizzle bug #4147:**
   Find the line that adds `account_id` to the `statements` table:
   ```sql
   ALTER TABLE "statements" ADD COLUMN "account_id" uuid;
   ```
   Then find the FK constraint for it. It should read:
   ```sql
   ALTER TABLE "statements" ADD CONSTRAINT "statements_account_id_financial_accounts_id_fk"
     FOREIGN KEY ("account_id") REFERENCES "public"."financial_accounts"("id") ON DELETE set null ON UPDATE no action;
   ```
   **VERIFY:** The `REFERENCES` clause says `"financial_accounts"` — NOT a garbled internal Drizzle name, NOT `__drizzle_migrations`, NOT an empty reference.

5. The `account_id` column has NO NOT NULL constraint (it must be nullable).
  </verify>
  <done>Human types "approved" confirming all 5 SQL checks pass, or "issues: [description]" if the FK reference is malformed — in which case migration must NOT proceed until the SQL is corrected.</done>
</task>
