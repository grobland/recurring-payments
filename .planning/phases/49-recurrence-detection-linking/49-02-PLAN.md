**Main function: `linkDetectedSeries(db: DbClient, userId: string, detectedSeries: DetectedSeries[]): Promise<LinkingResult>`**

Implementation steps per detected series:

**CRITICAL — DB pool constraint (per user decision in CONTEXT.md):** For each merchant group (each DetectedSeries item), wrap ALL write operations (steps 1-8 below) in a single `db.transaction(async (tx) => { ... })` call. Use the `tx` client for every insert, update, and query inside the transaction — do NOT use the outer `db` for writes within the loop. This minimizes connection usage against the Supabase free-tier 3-connection pool limit. The reads to find existing masters/links (steps 3-4) should also run inside the transaction to ensure consistency.

1. **Upsert recurring_series record.** Check if a series already exists for this merchantEntityId + userId. If exists: update stats (avgAmount, transactionCount, lastSeenAt, nextExpectedAt, confidence) rather than creating duplicate (LINK-06 idempotency). If new: insert. Use `onConflictDoUpdate` if possible, or select-then-upsert.

2. **Insert recurringSeriesTransactions junction rows** for each transactionId in the detected series. Use `onConflictDoNothing()` to handle reprocessing idempotently.

3. **Check for existing master link.** Query recurringMasterSeriesLinks for this series. If link already exists: preserve it (LINK-06). Check for amount changes (LINK-08):
   - If new avgAmount differs from master's expectedAmount by > 10%: create recurring_event with eventType "amount_changed" and metadata `{ oldAmount, newAmount, percentChange }`.
   - If change > 50%: also create a review_queue_item with itemType "amount_change".
   - Update master's lastChargeDate and nextExpectedDate.
   - Skip to next series.

4. **Find matching existing master (LINK-01 steps 2-3).** Query recurringMasters where merchantEntityId matches AND billingFrequency is compatible AND amount range overlaps. Compatibility: same frequency, or master has no frequency yet.

5. **Apply confidence thresholds:**
   - **>= 0.85 (LINK-02):** Auto-link. If matching master found (step 4): create recurringMasterSeriesLinks entry + recurring_event "master_auto_linked". If no master found: auto-create new recurring_master + link + recurring_event "created".
   - **0.60-0.84 (LINK-03):** Create review_queue_item with itemType "new_series", suggestedAction JSONB: `{ action: "link" | "create_new", targetMasterId: matchingMaster?.id, reason: "Detected by rule X with confidence Y" }`. Also create the series record but don't link to master.
   - **< 0.60 (LINK-04):** Series exists in DB but no master link, no review item. Just log the series.

6. **Auto-create master (when confidence >= 0.85, no existing master):**
   - name: detected series merchantName
   - recurringKind: inferred from heuristics (see inferRecurringKind below)
   - status: "active"
   - amountType, expectedAmount (= avgAmount), expectedAmountMin, expectedAmountMax, currency: from detected series
   - billingFrequency: detectedFrequency
   - billingDayOfMonth: dayOfMonth
   - nextExpectedDate: from detected series
   - lastChargeDate: lastSeenDate
   - confidence: from detected series
   - merchantEntityId: from detected series

7. **Account migration (LINK-09):** The detection engine already groups across all accounts (merchantEntityId is account-agnostic). When linking, if we find an existing master with the same merchantEntityId on the SAME user, link to it regardless of which account the transactions came from. Create recurring_event "account_migration" with metadata noting the account difference only if transactions span multiple accountIds (check via statements table).

8. **Audit trail (DETECT-09):** For every action, write to recurring_events:
   - "series_created": when new series record inserted
   - "master_created": when new master auto-created
   - "master_auto_linked": when series linked to existing master
   - "amount_changed": when amount differs > 10%
   - "review_item_created": when review queue item added
   - Metadata JSONB includes: `{ rule, confidence, transactionCount, amountRange: { min, max, avg }, seriesId }`

```typescript
export function inferRecurringKind(
  frequency: string,
  amountType: "fixed" | "variable",
  merchantCategory: string | null
): "subscription" | "utility" | "insurance" | "loan" | "rent_mortgage" | "membership" | "installment" | "other_recurring"
```
Heuristics:
- merchantCategory contains "utility/electric/gas/water/telecom/internet/phone" => "utility"
- merchantCategory contains "insurance" => "insurance"
- merchantCategory contains "loan/mortgage/rent" => frequency is monthly? "loan" or "rent_mortgage"
- amountType === "fixed" AND frequency is monthly AND category is entertainment/software/media/streaming => "subscription"
- amountType === "fixed" AND frequency is monthly AND category is gym/fitness/club => "membership"
- Default: "other_recurring"
